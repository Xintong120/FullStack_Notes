context_window_management
  • 提示压缩策略架构 (Prompt Compression Strategy Architecture)
    • 总体目标
      • 约束条件：token_count(prompt) + reserve ≤ target_tokens
      • 保护原则：首尾消息 + 工具调用消息优先保留
      • 压缩顺序：智能截断 → 消息删除 → 最终截断
      • 容错机制：lossy_ok参数控制失败处理
    • 策略1：Token-aware truncation (基于Token的智能截断)
      • 核心思路
        • 渐进式压缩：从粗到细逐步减小截断粒度
        • 内容优先：仅截断消息内容，保留消息结构
        • 范围控制：仅处理中间消息(1:-1)，保护首尾
        • Token精确：基于实际token计数进行截断
      • 执行逻辑
        • 初始化cap = start_cap (默认8192 tokens)
          • 设置初始截断上限为start_cap，默认8192个token
        • 循环条件：total_tokens() + reserve > target_tokens & cap ≥ floor_cap
          • 当总token数加上预留空间超过目标token数且截断上限不小于最低上限时继续循环
        • 处理对象：msgs[1:-1] (跳过首尾消息)
          • 处理除首尾消息外的所有消息
        • 判断标准：_tok_len(content) > cap
          • 如果消息内容的token长度超过截断上限
        • 截断方法：_truncate_middle_tokens(content, enc, cap)
          • 使用中间省略截断方法将内容压缩至指定token长度
      • 截断算法 (_truncate_middle_tokens)
        • 输入参数：text, tokenizer, max_tokens
        • 空间分配：head = max_tokens // 2 - 1
        • 保留策略：头部(head) + 中间省略号 + 尾部(tail)
        • 编码转换：使用tokenizer.encode/decode保持语义
        • 输出格式：head_content + " … " + tail_content
      • 动态调整机制
        • 指数衰减：cap //= 2 (每次减半)
        • 渐进收紧：8192 → 4096 → 2048 → 1024 → ...
        • 保底机制：floor_cap = 128 tokens
        • 自适应：根据实际压缩效果调整粒度
      • 优势特点
        • 精确控制：基于真实token计数而非字符数
        • 内容保留：保持消息结构和关键信息
        • 渐进优化：多次迭代达到最佳压缩效果
        • 语义友好：中间省略号保持可读性
      • 局限性
        • 时间复杂度：多次token计算开销较大
        • 适用范围：仅适用于内容过长的消息
    • 策略2：Middle-out deletion (中间向外删除)
      • 核心思路
        • 空间优先：当内容截断不足时，删除整个消息
        • 对称删除：从对话中间开始向两侧扩展
        • 智能保护：保留工具调用相关的关键消息
        • 连贯性保证：保持对话的逻辑完整性
      • 执行逻辑
        • 初始化：centre = len(msgs) // 2
        • 循环条件：total_tokens() + reserve > target_tokens & len(msgs) > 2
        • 删除顺序：构建对称的中心向外索引序列
          • 第一轮：删除centre位置消息
          • 第二轮：centre+1 和 centre-1
          • 后续轮：继续向外扩展
        • 保护机制：跳过tool_calls和role=="tool"的消息
        • 终止条件：无更多可删除消息或达到token预算
      • 删除顺序算法
        • 基础索引：[centre]
        • 对称扩展：zip(range(centre+1, len-1), range(centre-1, 0, -1))
        • 展开序列：[centre, centre+1, centre-1, centre+2, centre-2, ...]
        • 实际效果：从对话核心向边缘删除，保持首尾完整
      • 消息保护机制
        • 工具调用保护："tool_calls" in msg
        • 工具响应保护：msg.get("role") == "tool"
        • 首尾保护：msgs[0] 和 msgs[-1] 永不删除
        • 连贯性保证：保护对话的关键逻辑节点
      • 优势特点
        • 高效压缩：删除整个消息释放更多空间
        • 语义保护：保留工具调用的上下文完整性
        • 对称平衡：保持对话的左右对称性
        • 渐进安全：每次只删除一条消息便于控制
      • 局限性
        • 信息丢失：删除整个消息可能丢失有用信息
        • 保护开销：每次循环都要检查消息类型
    • 策略3：Last-chance trim (最终安全网截断)
      • 核心思路
        • 最后手段：当前面策略都无法达到预算时启用
        • 重点保护：仅对首尾消息进行强制截断
        • 最小破坏：保持对话的开始和结束部分完整
        • 兜底机制：确保无论如何都能满足token预算
      • 执行逻辑
        • 初始化：cap = start_cap (重新开始截断循环)
        • 循环条件：total_tokens() + reserve > target_tokens & cap ≥ floor_cap
        • 处理对象：仅msgs[0]和msgs[-1] (首尾消息)
        • 判断标准：_tok_len(text) > cap
        • 截断方法：_truncate_middle_tokens(text, enc, cap)
        • 收紧策略：cap //= 2 (指数衰减)
      • 为什么只处理首尾消息
        • 对话重要性：首消息包含用户意图，最后消息包含最新状态
        • 连贯性保证：保护对话的核心逻辑链条
        • 最小影响：相比删除中间消息，截断首尾消息内容的影响相对较小
        • 语义完整：保持对话的开头和结尾语义
      • 与策略1的区别
        • 适用范围：策略1处理所有中间消息，策略3只处理首尾
        • 触发时机：策略1是第一道防线，策略3是最后防线
        • 破坏程度：策略3更激进，可能破坏重要内容
        • 保护级别：策略3不保护任何消息，强制压缩至预算
      • 优势特点
        • 兜底保障：确保一定能满足token预算要求
        • 最小影响：只截断最不重要的首尾消息
        • 简单有效：逻辑简单，执行效率高
        • 完全控制：通过floor_cap参数控制最小可接受长度
      • 局限性
        • 破坏性：可能截断重要信息
        • 不可逆：一旦截断无法恢复
        • 语义风险：可能破坏对话的关键表达
